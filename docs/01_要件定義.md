# AI部署運営システム「DEPT-OPS」要件定義書

作成日: 2026-07-07 / 発注者: とみぃ（株式会社東京フラワー / 株式会社GSD）

## 1. 目的

各部署（AIエージェント）が毎日自律的に自社ECの運営状況を調査・報告・提案し、
人間（とみぃ）は「承認するかどうか」の判断だけに集中できる状態をつくる。

- 報告・提案までは**全自動**
- 書き込み系の実行は**承認制**（第1弾は承認キューへの登録まで。実行部は第2弾）

## 2. 第1弾で稼働する部署

| ID | 部署名 | データソース | 主な仕事 |
|----|--------|-------------|----------|
| market | マーケティング部 | RMS CouponAPI 2.0 / ItemAPI 2.0 | クーポン設定状況・期限切れ間近の検知、イベント対応漏れの指摘、施策提案 |
| seo | SEO室 | Search Console API（webmasters.readonly）+ 前回提案履歴 | 検索パフォーマンス変化の検知、**前回提案が実行されたかの検証**、新提案 |
| items | 商品管理部 | RMS ItemAPI 2.0（全商品スキャン） | 在庫切れ・説明文異常・画像欠落・薬機法リスク語の検出、修正提案 |

部署は「部署定義」シートの1行として定義し、行を追加すれば部署が増える設計とする
（役割プロンプト・データソース・有効フラグをシートで管理）。

## 3. アーキテクチャ

```
[GAS 時間主導トリガー 毎朝6:00]
        │
   orchestrator.gs ── 部署定義シートを読み、有効な部署を順番に実行
        │
   ├─ collector（部署ごと）: RMS / GSC からデータ取得
   ├─ claude.gs: 役割プロンプト + 今日のデータ + 前回報告 → Claude API → JSON応答
   │     { status, headline, report, proposals[], needs_decision }
   ├─ 報告履歴シートに保存（部署ごとの履歴 = 次回の入力）
   ├─ notify.gs:
   │     ・報告 → Chatwork（全部署まとめて朝会形式）
   │     ・要承認の提案 → LINE WORKS（承認/却下リンク付き）
   └─ 承認キューシートに提案を登録
        │
[GAS doGet] ── JSON配信 ──> [GitHub Pages ダッシュボード]
[GAS doGet?action=approve/reject&id=..&sig=..] ── HMAC署名付き承認リンク
```

- スタック: GAS + Google Sheets + GitHub Pages + Claude API（既存の運勢神社・管理会計ダッシュボードと同構成）
- Claude モデル: 日次実行は claude-haiku-4-5（コスト最小）、必要に応じてシートで部署ごとに上書き可

## 4. スプレッドシート構成（1ブック）

| シート名 | 列 |
|---------|-----|
| 部署定義 | id, 部署名, 有効, モデル, 役割プロンプト |
| 報告履歴 | timestamp, dept_id, status(green/yellow), headline, report, proposals_json |
| 承認キュー | id, created, dept_id, 提案内容, status(pending/approved/rejected), decided_at |
| 実行ログ | timestamp, level, message |

## 5. 通知仕様

- **Chatwork（報告）**: 毎朝1メッセージ。`[info][title]🏢 朝の部署報告 YYYY-MM-DD[/title]🟢マーケ部: … 🟡SEO室: …[/info]` 形式
- **LINE WORKS（承認）**: needs_decision=true の提案ごとに1通。承認・却下のHMAC署名付きURLを添付（有効期限72h）
- LINE WORKS は Service Account JWT（既存の週次売上レポートと同方式）

## 6. ダッシュボード（GitHub Pages）

- 部署ごとのカード: 信号色（🟢/🟡）、今日のheadline、報告本文、提案一覧
- 承認キューの状況（pending件数）
- 履歴ビュー（部署選択で過去の報告を遡れる）
- GAS doGet の JSON を fetch。閲覧はURL秘匿+読み取り専用（第2弾でOTP認証を管理会計ダッシュボードから移植可）

## 7. セキュリティ

- 認証情報はすべて Script Properties（コードに直書き禁止）
  - `RMS_SERVICE_SECRET`, `RMS_LICENSE_KEY`, `ANTHROPIC_API_KEY`,
    `CHATWORK_TOKEN`, `CHATWORK_ROOM_ID`,
    `LW_CLIENT_ID`, `LW_CLIENT_SECRET`, `LW_SERVICE_ACCOUNT`, `LW_PRIVATE_KEY`, `LW_BOT_ID`, `LW_CHANNEL_ID`,
    `HMAC_SECRET`
- 承認リンクは HMAC-SHA256 署名 + 有効期限で改ざん・使い回しを防止
- Search Console は appsscript.json の oauthScopes に webmasters.readonly を追加し ScriptApp.getOAuthToken() を使用

## 8. 段階リリース

1. **Phase 1（本パッケージ）**: 読み取り→報告→提案→承認キュー登録まで
2. **Phase 2**: 承認済み提案の自動実行（クーポン発行・バナー更新など既存RMS自動化資産を実行部として接続）
3. **Phase 3**: 部署追加（経理部=freee、SNS部=Metricool）、ダッシュボードOTP認証

## 9. 完成の定義（Definition of Done）

acceptance_criteria.md の全項目がPASSし、
実データでの1サイクル（収集→報告→CW通知→LW承認リンク→ダッシュボード表示）が通ること。
