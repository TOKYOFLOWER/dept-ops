# 受け入れ基準（acceptance_criteria）

すべて PASS で完成。テスト方法と結果は docs/TEST_RESULTS.md に記録すること。

## A. オーケストレーター

- [ ] A1: 部署定義シートで 有効=TRUE の部署だけが実行される（無効部署はスキップされログに残る）
- [ ] A2: 1部署でエラーが起きても残りの部署は実行され、エラーは実行ログシートとChatworkに通知される
- [ ] A3: 各部署の実行で「前回の報告履歴（最新1件）」がプロンプトに含まれる

## B. Claude応答

- [ ] B1: 応答は必ず JSON にパースできる（```json フェンス付き応答でも成功する）
- [ ] B2: パース失敗時は1回リトライし、それでも失敗なら status=yellow の定型報告でフォールバック
- [ ] B3: JSONに status / headline / report / proposals / needs_decision の全キーが揃う（欠損時はデフォルト補完）

## C. 通知

- [ ] C1: Chatwork報告は3部署分が1メッセージにまとまり、info/title 記法で整形される
- [ ] C2: needs_decision=true の提案がある場合のみ LINE WORKS に承認依頼が送られる
- [ ] C3: LINE WORKS の JWT が生成でき、アクセストークン取得→メッセージ送信が成功する（ドライラン関数あり）

## D. 承認フロー

- [ ] D1: 承認リンクの HMAC 署名が不正な場合は 403 相当のエラーページを返す
- [ ] D2: 発行から72時間を過ぎたリンクは「期限切れ」を表示し状態を変更しない
- [ ] D3: 既に approved/rejected の提案に再アクセスしても状態が上書きされない
- [ ] D4: 承認/却下の結果が承認キューシートに decided_at 付きで記録される

## E. ダッシュボード

- [ ] E1: doGet(action=data) が最新報告・承認キュー・履歴30件を含むJSONを返す
- [ ] E2: 3部署のカードに信号色・headline・報告・提案が表示される
- [ ] E3: 部署を選ぶと過去の報告履歴を遡って読める
- [ ] E4: スマホ幅(375px)でレイアウトが崩れない
- [ ] E5: データ取得失敗時にエラーメッセージと再読込ボタンが表示される
- [ ] E6: 今朝のサマリーバーが表示され、全部署の状態（🟢/🟡部署数）と承認待ち件数が1行の文字情報として分かる（3秒で温度感を把握できる）
- [ ] E7: 部署カードが3層構造になっている。headlineは常時表示、根拠（report本文＋データ取得時刻）は折りたたみ表示、今日のアクションは承認状態に応じて出し分ける（承認済み=「✅実行OK：〜」／承認待ち=「⏳社長の承認待ち」／却下は非表示、承認不要の提案はそのまま表示）
- [ ] E8: CTR・掲載順位・表示回数・薬機法リスク語・在庫API等の専門用語にタップで開く短い説明が表示される（title属性のみに依存しない）
- [ ] E9: 承認キューをpending/approved/rejectedタブで切り替えて一覧表示できる（各行に部署名・提案タイトル・決裁日時、pendingには「社長のLINE WORKSに依頼済み」の注記）。各部署カード上部に直近の状況推移がドット列（🟢🟡＋テキストラベル）で表示される。375px幅で崩れず、色だけに依存しない表示になっており、フォントサイズは14px以上を保つ
